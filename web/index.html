About Us

OpsLens AI is an incident investigator for Kubernetes teams

We built OpsLens to help operators understand why a Kubernetes service failed without jumping between kubectl outputs, logs, events, metrics, and scattered notes.

Goal

Turn operational noise into a clear root-cause story with evidence, recommended action, and verification steps.

01

The problem we solve

A production incident rarely points to one obvious source. A Service may have no endpoints, a Pod may fail, logs may show connection errors, and metrics may show pressure at the same time. OpsLens connects those signals and separates the real cause from secondary findings.

02

What we provide

OpsLens provides a focused investigation workflow: scope selection, evidence collection, specialized agents, Supervisor correlation, LLM explanation, safe remediation commands, and exportable incident reports.

03

Why it matters

The user does not need to manually connect every Kubernetes symptom. OpsLens explains what failed, why it likely failed, what evidence supports that conclusion, and what should be checked next.

Help Center

How to use OpsLens

Follow these steps when testing a scenario or presenting the application.

1. Start your cluster

Make sure Minikube is running and kubectl can read nodes, namespaces, and pods.

2. Open Investigate

Select a node first, then choose a namespace that has Pods running on that node.

3. Choose a scenario

Apply an incident scenario when you want a controlled test case for the presentation.

4. Launch analysis

OpsLens runs collectors, agents, Supervisor reasoning, and report generation.

5. Review the report

Read the problem, evidence, root cause, recommended fix, and verification plan.

6. Export result

Download the report as PDF, Excel, Markdown, or JSON for submission or presentation.

Common errors

No nodes found

Run minikube status and kubectl get nodes.

No namespaces for selected node

Check kubectl get pods -A -o wide and confirm Pods are scheduled on that node.

Scenario did not apply

Check the FastAPI terminal logs and verify the scenario YAML exists.

LLM report failed

Make sure GEMINI_API_KEY is set before starting uvicorn.
